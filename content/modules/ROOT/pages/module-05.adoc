= Section 5 — Automated incident response (Splunk → EDA → Vault)
:icons: font
:idprefix:
:idseparator: -

Companion to `section5/README.md`. Brute-force simulation → Splunk detection → webhook → EDA → AAP job revokes app DB credentials.

include::../../docs/lab/common-environment.adoc[]

[IMPORTANT]
.Assume breach → contain blast radius
====
Detection will always be imperfect. Zero Trust asks **what happens next**. Here, a high-volume SSH failure pattern triggers **automated revocation** of the application’s **database credentials**, not just an email. This limits what an attacker who gained foothold on the app host can reach next (lateral movement to sensitive data). The app may go **intentionally unhealthy**. You’re trading availability for confidentiality in the critical window after suspected compromise.
====

== Prerequisites

* Section 2 complete (app deployed with Vault DB creds).
* Splunk receiving `linux_secure` from app/db (`setup/integrate-splunk.yml` as deployed).
* **Event-Driven Ansible controller** (AAP 2.6) receiving webhooks, or standalone `ansible-rulebook` for demos.
* Section 5 EDA resources deployed (`configure-aap-project.yml --tags eda,section5,rbac`): EDA project, Decision Environment, **ZTA EDA Event Stream** credential (token from Vault).

[TIP]
.Visible telemetry
====
Splunk indexes **evidence** from hosts (via forwarders). ZTA monitoring isn't optional decoration. It feeds **continuous security analytics** that can drive **orchestrated response** through EDA without waiting for humans to notice a dashboard.
====

== Exercise 5.1 — AAP job templates

[cols="2,3",options="header"]
|===
| Template | Playbook
| Emergency: Revoke App Credentials | `section5/playbooks/revoke-app-credentials.yml`
| Simulate Brute Force | `section5/playbooks/simulate-bruteforce.yml`
| Restore App Credentials | `section5/playbooks/restore-app-credentials.yml`
|===

Use **ZTA Lab Inventory** and **ZTA Machine Credential**.

[TIP]
.Revoke vs “block IP”
====
This lab **revokes secrets** in Vault (and stops reliance on DB access) instead of only blocking a source IP. Useful when attackers pivot internally or use many IPs. The approach: **credential-centric containment**. Shrink **privilege** first; network blocks can layer on top.
====

== Exercise 5.2 — Splunk saved search / alert

In Splunk (`http://central.zta.lab:8000`), confirm failed SSH events exist:

[source]
----
index=zta_app sourcetype=syslog "Authentication failure"
----

Also check VM-level auth logs:

[source]
----
index=zta_syslog sourcetype=syslog sshd "Authentication failure"
----

Create alert (example search from README):

[source]
----
(index=zta_app OR index=zta_syslog) sourcetype=syslog "Authentication failure"
| stats count by src_ip, host
| where count >= 5
----

Real-time window (e.g. 60s), webhook action to EDA: `http://control.zta.lab:5000/endpoint` with auth token `zta-eda-webhook-a1b2c3d4-e5f6-7890` (from Vault KV `secret/services/eda-webhook`; must match the **ZTA EDA Event Stream** credential).

REST API variant documented in `section5/README.md`.

[TIP]
.SIEM as sensor, EDA as actuator
====
Splunk answers “**do we see attack-like behaviour?**” The webhook sends a **standardised signal** into **Event-Driven Ansible**, which turns that boolean into **approved automation** (run job template). The ZTA pattern here: **policy-driven orchestration**. Response playbooks are still **subject to automation controller** policies and RBAC, not anonymous shell scripts.
====

== Exercise 5.3 — EDA rulebook

* **Event-Driven Ansible controller (AAP 2.6):** The EDA project, Decision Environment, and credentials were created by the prerequisite setup. Create a **Rulebook Activation**: name `Splunk Brute Force Response`, rulebook `splunk-credential-revoke`, Decision Environment `ZTA Decision Environment`, credential **ZTA EDA Event Stream** (token-based webhook auth from Vault), restart policy `On failure`. Enable the activation — it listens on port 5000. See https://docs.redhat.com/en/documentation/red_hat_ansible_automation_platform/2.6/html/using_automation_decisions/[Using automation decisions].
* CLI example (no controller):

[source,bash]
----
ansible-rulebook --rulebook extensions/eda/rulebooks/splunk-credential-revoke.yml \
  -i inventory/hosts.ini --verbose
----

[TIP]
.Closed loop
====
EDA closes the loop between **observation** (logs/alerts) and **enforcement** (AAP). For Zero Trust maturity, measure **time-to-contain** and ensure playbooks are **idempotent** and **safe**. False positives will happen, which is why Section 5.8 tuning matters.
====

== Exercise 5.4 — Baseline health

[source,bash]
----
curl -s http://app.zta.lab:8081/health
# On DB: confirm Vault dynamic role exists
----

[TIP]
.Know “normal” before chaos
====
Baseline health and DB role presence give you **before** pictures. Incident response without baselines turns into guesswork. ZTA operations emphasize **continuous visibility** so automated actions are explainable to auditors (“we knew the app had active creds X; after revoke, creds were gone”).
====

== Exercise 5.5 — Simulate attack

Launch **Simulate Brute Force** (or `ansible-playbook section5/playbooks/simulate-bruteforce.yml`).

[TIP]
.Safe simulation
====
The playbook generates **failed auth** events **without** granting access. Useful training data for Splunk thresholds and for proving the **detection → response** chain without handing students real exploit tools.
====

== Exercise 5.6 — Observe chain

* Splunk: alert fired, events visible.
* EDA: event received, rule matched, `run_job_template` to revoke.
* Automation controller: **Emergency: Revoke App Credentials** ran without manual launch.
* App/CURL and PostgreSQL: creds gone / app isolated.

[TIP]
.Human-in-the-loop recovery
====
Automation contained the incident. **Humans** still decide when to restore, matching many real SOCs (investigate, then re-issue least privilege). The **restore** template models **controlled re-onboarding** of credentials rather than silently rolling back security.
====

== Exercise 5.7 — Restore

Run **Restore App Credentials**; confirm `health` OK.

[TIP]
.Re-verify after recovery
====
Post-restore checks prove **new** credentials and **clean** state, not the old lease. Think of it as **re-authorization** after an incident: you don't assume the previous session is still trustworthy.
====

== Validation checklist

* [ ] Pre-attack healthy
* [ ] Splunk alert + webhook + EDA + automation controller revoke job
* [ ] Post-attack app isolated; restore succeeds
