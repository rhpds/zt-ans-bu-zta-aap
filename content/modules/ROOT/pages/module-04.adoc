== Architecture summary

* **Outer ring:** user must be in `network-admins` to launch VLAN/Network templates (see `aap.gateway` in deployment).
* **Inner ring:** trusted SPIFFE ID, `network-admins`, VLAN 100–999, allowed action.

Trusted workload SPIFFE ID (lab): `spiffe://zta.lab/workload/network-automation`

[IMPORTANT]
.Why two rings?
====
**Humans** can be phished; **scripts** can be stolen. The **outer ring** ensures only the right *people* can press “go” on high-risk network automation. The **inner ring** adds **workload identity**: the job proves it runs on **the attested automation platform** (SPIRE SVID), not someone’s laptop running a copy of the playbook. Together they implement **defence in depth**—an attacker needs both a valid user session *and* a trusted execution environment path to change the fabric.
====

[TIP]
.CMDB as audit and SoT
====
NetBox updates record **who** initiated change and **which workload identity** was bound to the automation. That is invaluable for **forensics** (“was this VLAN created by a rogue job?”) and for keeping **inventory trustworthy** for the next policy decision.
====

== Exercise 4.2 — Denied (`neteng`)

Follow `section4/README.md`: launch as `neteng` and observe SPIFFE verification vs policy outcome for your deployed gateway rules.

[TIP]
.User vs workload signals
====
You may see **SPIFFE verified** while the **human** lacks `network-admins`—a deliberate teaching point: **trust the platform** and **authorise the person** are separate questions. Alternatively, if your gateway blocks `neteng` earlier, compare that to the README narrative: both patterns exist in real deployments depending how `aap.gateway` is scoped.
====

== Exercise 4.3 — Allowed (`netadmin`)

As `netadmin`, create VLAN 200 (DMZ). Verify on Arista (`show vlan brief`) and in NetBox (audit fields).

[TIP]
.Segmentation under policy control
====
VLANs are a building block of **micro-segmentation**. Creating them only through gated automation reduces **untracked lateral movement** helpers (shadow VLANs) and ties network change to **identity + workload proof**.
====

== Exercise 4.4 — Edge cases

* Invalid VLAN `5000` → OPA deny (outside range).
* Stop **spire-agent** on `control` → SPIFFE step fails; restart after test.
* Optional: `ipa group-add-member network-admins --users=neteng` then retry; clean up membership afterward.

[TIP]
.Parameter validation at runtime
====
Invalid VLAN IDs and **missing SVIDs** are caught **after** launch but **before** wide blast radius on the network—runtime PDP inputs include data the gateway did not know yet. That is why organisations combine **coarse** platform gates with **fine** in-playbook policy.
====

== Playbook flow (summary)

SPIFFE verification on AAP → OPA decision on central → Arista VLAN module → NetBox update (see README diagram).

== Discussion prompts

Why verify workload and user? What breaks if SPIRE is down? How does NetBox auditing help?

[TIP]
.Facilitator notes
====
* **User without workload trust** → stolen credentials are not enough if SPIRE is healthy and policy binds both.
* **SPIRE down** → automation cannot prove workload identity; **fail closed** is preferable to accepting unsigned requests.
* **NetBox** → ties human + workload metadata to **configuration items** for compliance and incident response.

== Validation checklist

* [ ] SPIRE on `control` yields expected SVID / playbook verification step succeeds
* [ ] Unauthorised path documented in job output matches policy intent
* [ ] `netadmin` success: VLAN on switches + NetBox
* [ ] Out-of-range VLAN denied
* [ ] SPIRE agent stop causes verification failure
